TEC technical challenge plan

Three Lambdas, one queue, one cache.

Fetch MISO on a schedule, buffer raw snapshots in SQS, write PostgreSQL through RDS Proxy, and serve external reads through API Gateway cache before Lambda ever touches the database.

01

Chosen Shape

Application

ASP.NET Core Minimal API

Small read surface, thin route handlers, service classes for ingestion and querying. Controllers would be extra ceremony for this scope.

Ingestion

Fetch Lambda -> SQS -> writer Lambda

Fetching MISO and writing PostgreSQL are different failure domains. SQS preserves the fetched payload if the database path is temporarily unavailable.

Database

Cache first, proxy second, DB last

API Gateway cache absorbs repeated reads. RDS Proxy pools real database connections. Unique keys keep duplicate queue deliveries harmless.

06

Interview Defense

"I separated fetch, write, and read because they fail and scale differently. The queue keeps a fetched MISO snapshot alive even when Postgres is unavailable." Answer for: why SQS?
"The database enforces idempotency. Code can have bugs; unique keys and transactions are the invariant. Duplicate SQS deliveries update the same tuples instead of appending new ones." Answer for: how do you prevent duplicates?
"The read API is cached because the feed changes at most once per minute. A 15-30 second TTL stops repeated latest reads before Lambda or Postgres are involved." Answer for: why cache?
"RDS Proxy is not the only guard. API Gateway cache, usage-plan throttles, Lambda concurrency, RDS Proxy limits, and bounded SQL all protect the database together." Answer for: how do you avoid flooding DB connections?
